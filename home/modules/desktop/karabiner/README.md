Karabiner-Elements is installed by the homebrew cask, not nixpkgs. nix-darwin's old `org.nixos.start_karabiner_daemons` and `org.nixos.setsuid_karabiner_session_monitor` launchd entries from prior `services.karabiner-elements.enable = true` packaging conflict with the cask-managed `org.pqrs.*` services and are removed on every home-manager activation by the orphan-cleanup script, which gates `sudo` behind presence checks so fresh machines incur zero privileged calls. The cask is declared in `hosts/macbook/karabiner/homebrew-cask.nix` because `homebrew.casks` is a nix-darwin system-level option and cannot sit inside a home-manager module; the same file asserts at activation that the installed version is at least 16.0.0 (the minimum for `to.send_user_command`, the rest of the module assumes it) and fails the rebuild with an upgrade hint if older.

Configuration is generated from `rules/default.nix` (Nix expressions → JSON via `builtins.toJSON`) and copied — not symlinked — to `~/.config/karabiner/karabiner.json` at activation. Karabiner-elements [does not detect config changes when the file is a symlink][config-file-path]. After every copy the per-user `org.pqrs.service.agent.karabiner_console_user_server` is kickstarted (the [documented reload mechanism][config-file-path]) so new rules take effect immediately without a GUI relaunch.

`karabiner_console_user_server` has a long-standing upstream bug where it does not reliably recover after macOS sleep/wake cycles ([issues #1932][issue-1932], [#3078][issue-3078], among others). The `restart-on-wake` daemon fills the gap: it kicks `console_user_server` unconditionally on `NSWorkspaceDidWakeNotification`, kicks reactively when `karabiner_cli` IPC probes fail consecutively (the IPC probe roundtrips through `console_user_server` itself — the same component that degrades — so a failed probe is evidence rules may also stop firing), and kicks on a periodic safety-net interval to catch failure modes that degrade rule firing without breaking the IPC channel. Each kick is `launchctl kickstart -k` (SIGKILL + launchd respawn), so high `runs` counts on the launchd job are expected and not a sign of upstream instability — they are the watchdog doing its job. Exact intervals and thresholds live as named constants at the top of `restart-on-wake/scripts/karabiner-restart-on-wake-daemon`.

An earlier active canary tried CGEventPost-ing an F19 keystroke and listening for the matching karabiner rule fire, but was removed because macOS gates keyboard event posting from launchd-managed processes behind Accessibility permission, which is bound to a specific binary identity and cannot be granted persistently to a daemon whose nix store path changes on every rebuild. `karabiner_cli` is a stable-path binary shipped by Karabiner-Elements itself, requires no permissions, and exercises the same IPC channel rules flow through — so it replaced the canary while preserving the end-to-end "is karabiner actually working" signal.

The daemon writes a current-state snapshot to `/tmp/karabiner-health.json` and one JSON event per line to `/tmp/karabiner-daemon.log`. The `karabiner-status` CLI shipped by this module reads the snapshot and prints a human summary; it accepts `--json` for raw output and `--log-tail N` for tailing the event log, and its exit code is script-friendly (zero when both karabiner processes are running and the last IPC probe succeeded, non-zero otherwise).

The workspace-window-switcher Cmd+Tab keybinding is the only rule that does not use `to.shell_command`. It uses `to.send_user_command` ([v16.0.0 addition][send-user-command]) which sends a UNIX datagram directly from `karabiner_console_user_server` to the listening daemon at `/tmp/workspace-switcher.sock`. The shell_command path fork+execs `/bin/sh -c <client> next` per keypress, which dominated the latency budget under CPU load and was the documented cause of missed keystrokes. send_user_command is sub-millisecond and survives heavy load. The daemon side lives at `hosts/macbook/scripts/workspace-window-switcher-daemon-swift-sources/` and binds the socket as `SOCK_DGRAM`. Other rules continue to use `shell_command` because they trigger long-running actions (aerospace, summon-browser, etc.) where fork+exec cost is irrelevant.
